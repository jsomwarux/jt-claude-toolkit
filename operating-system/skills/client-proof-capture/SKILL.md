# Client Proof Capture

Use when client work can become proof without violating privacy or trust.

## Required Gates
Do not publish, pitch, or reuse proof until all relevant gates are explicit:
- accepted deliverable or client-verified output
- payment or scope status
- proof-safe evidence captured
- permission or anonymization path
- client-sensitive data removed
- metric or outcome named
- reusable workflow/IP pattern captured

## Workflow
1. Identify client, deliverable, status, and buyer/operator.
2. Check the client folder and Client OS. If missing for an active client, initialize the Client OS template when available.
3. Capture evidence paths: screenshots, redacted samples, run logs, acceptance notes, metrics, edge cases, failures, judgment calls.
4. Route outputs to proof checklist, reusable IP/failure/automation logs, portfolio queue, content proof point, or referral ask only when gates are satisfied.

## Privacy Rule
Client proof defaults to private. Public naming requires explicit permission. Otherwise use anonymized, redacted, or synthetic examples.

## Output
Return proof status: `blocked`, `private-ready`, or `publish-ready`; missing gates; files updated; recommended next action.
